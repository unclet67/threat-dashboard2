Architecture Approaches

Three ways to structure this app. Pick the one that fits your team's setup best.

A

Full-stack Monorepo — React + Express + SQLite (Recommended)

React SPA frontend · Express REST API backend · SQLite database · All running together on one machine

Browser → React UI → Express API :3001 → SQLite .db file
Express API → Claude API + MITRE ATT&CK

Pros

  • Proper relational data model
  • API key stays on server (secure)
  • One instance shared by whole team
  • Everything in JS/TS

Cons

  • Two processes to run (Vite + Express)
  • Slightly more setup
B

Static SPA + JSON Files

Pure React SPA · Knowledge base stored as structured JSON files · Claude API called directly from browser

Browser → React UI → JSON data files + Claude API (direct)

Pros

  • No backend to run or maintain
  • Data is git-friendly (plain files)
  • Simplest possible setup

Cons

  • API key exposed in browser
  • Complex relationships get messy in JSON
  • No real querying across actors/events
  • Concurrent team edits cause conflicts
C

React + PocketBase

React SPA frontend · PocketBase as backend (single binary, no config) · Built-in admin UI for managing data

Browser → React UI → PocketBase :8090 → SQLite (built-in)
+ Small Node sidecar for Claude API proxy

Pros

  • Free admin dashboard for data entry
  • Real-time subscriptions built in
  • Single binary, minimal setup

Cons

  • Go binary dependency
  • Still needs Node sidecar for Claude
  • Mixed ecosystem (Go + JS)
  • Less control over API design
